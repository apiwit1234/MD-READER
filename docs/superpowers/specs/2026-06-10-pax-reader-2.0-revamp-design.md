# PAX Reader 2.0 Revamp — Design Spec

Date: 2026-06-10
Status: Approved by user (pending spec review)

## Goal

Revamp PAX Reader (Electron + Next.js markdown reader) in three areas while
preserving every existing feature (multi-folder navigation, tabs, git panel,
terminal, search, fonts, zoom, UI-size scaling, settings, file associations):

1. Replace full-reinstall NSIS updates with Velopack delta (patch) updates.
2. Make mermaid rendering reliable and real-time on file save; support
   standalone `.mmd`/`.mermaid` files.
3. Fresh Canva-inspired design system with 6 themes (default Dark + Light,
   plus Aurora, Nord, Sepia Paper, Dracula). Cartoon themes are retired.

Implementation order (independently shippable): Mermaid → UI/Themes → Velopack.

---

## Part 1 — Update system: Velopack delta updates

### Current state

- `electron-updater` in `electron/main.cjs:109-166`; NSIS full installer is
  downloaded (~100MB) and silently re-run on quit (a reinstall).
- Targets: nsis + portable + zip (`package.json` build.win.target).
- Renderer surface: `update:check` / `update:install` IPC +
  `update:onUpdateReady` event; Settings → Updates section.

### Target state

**Packaging (developer side only):**

- `electron-builder --dir` continues to produce the unpacked app under
  `release/win-unpacked` (keeps current `files`/`asarUnpack` config so
  node-pty native binaries ship unchanged).
- `vpk pack` wraps that directory into Velopack artifacts: full `.nupkg`,
  delta packages, self-contained `Setup.exe` (renamed
  `PAX-Reader-Setup.exe`), and `RELEASES` manifest.
- `vpk upload github` publishes artifacts to the existing
  `apiwit1234/MD-READER` GitHub Releases repo.
- New npm scripts: `release:pack` (build + vpk pack) and `release:publish`
  (pack + upload). Developer runs one command; end users never run any.
- Portable .exe target stays (built by electron-builder as today) for
  no-install use; portable builds never self-update (current behavior).
- NSIS and zip targets are removed after the migration release.

**Runtime (main process):**

- Add `velopack` npm package to `dependencies` (main-process dep, same
  category as node-pty).
- `VelopackApp.build().run()` is the first statement in `electron/main.cjs`
  (required: handles install/uninstall/update hooks and may exit early).
- New `electron/updater.cjs` module replacing the electron-updater block:
  - `UpdateManager` with `GithubSource('https://github.com/apiwit1234/MD-READER')`.
  - `checkForUpdates()` → returns `{ available, version, notes }`.
  - `downloadUpdates(progressCb)` → emits percent progress.
  - `applyUpdatesAndRestart()` / `waitExitThenApplyUpdates()` for
    install-on-quit.
  - No-op guard when not packaged (dev mode) or portable.

**IPC surface (preload.cjs):**

- Keep `update:check`, `update:install`, `update:onUpdateReady`.
- Add `update:download`, `update:onProgress` (percent), `update:onAvailable`
  (`{version, notes}`).

**Renderer UX (Settings → Updates, `src/components/settings/UpdatesSection.tsx`):**

- "Check for updates" button → shows result: up to date, or new version +
  release notes.
- Download progress bar while downloading.
- "Restart & Update" button → applies delta patch and relaunches.
- Toggle (default ON, persisted in settings.json as today): auto-check on
  launch, auto-download, install on quit. Toast "Update vX.Y.Z ready —
  restart to apply" preserved.
- All update actions are UI-only. End users need no prerequisites: Velopack
  Setup.exe is self-contained native code (no Node, no .NET runtime, no
  admin rights).

**Migration for existing NSIS installs (one-time, UI-only):**

1. Release `v1.1.0` publishes BOTH artifact sets: a final NSIS installer
   (so current installs auto-update one last time via electron-updater) and
   the Velopack artifacts.
2. The `v1.1.0` app detects its install type at startup: NSIS install
   (running from a path containing the NSIS install dir / uninstaller
   present) vs Velopack install (Velopack hooks fired).
3. On an NSIS install, show a one-time dialog: "PAX Reader is switching to
   a faster update system — Upgrade now". On click: download
   `PAX-Reader-Setup.exe` from the same release, run it silently, then run
   the old NSIS uninstaller silently (`Uninstall PAX Reader.exe /S`),
   preserving `%APPDATA%` settings (uninstaller already doesn't touch them).
   Relaunch into the Velopack install.
4. `v1.1.1+` releases publish Velopack artifacts (+ portable) only.

**Accepted trade-offs:**

- Install dir becomes `%LOCALAPPDATA%\PAX-Reader` (no choose-folder).
- Settings/fonts/logs in `%APPDATA%` are untouched by install/update/migrate.

### Error handling

- Update check failures: logged to userData/logs (existing log-files.cjs),
  silent on auto-check, surfaced inline in Settings on manual check.
- Download/apply failures: Velopack falls back from delta to full package
  automatically; if apply fails the existing install keeps running (Velopack
  applies atomically on restart).
- Migration dialog failure path: keep running on NSIS install; offer retry
  on next launch; never uninstall before the new install verified (Setup
  exit code 0).

### Testing

- Unit: updater.cjs guards (dev/portable no-op), install-type detection.
- Manual release-candidate checklist (documented in docs/RELEASING.md
  rewrite): pack v0 → install → pack v1 → in-app check shows v1 → delta
  download size sanity (<20MB) → restart applies → settings preserved →
  terminal (node-pty) works post-update.

## Part 2 — Mermaid: reliability + real-time

### Current state

- Mermaid only renders from ```mermaid fences in .md via
  `src/components/MarkdownView.tsx:89-97` → `MermaidBlock.tsx`.
- No standalone `.mmd`/`.mermaid` support.
- File changes route through a folder-wide recursive watcher with 1200ms
  debounce (`electron/main.cjs:371-419`) + full cache invalidation.
- Failures: parse errors can render nothing useful; entity-normalization
  retry exists (`src/lib/mermaid-entities.ts`); errors logged to file only.

### Target state

**Standalone diagram files:**

- `.mmd` and `.mermaid` files appear in the sidebar tree and open as
  first-class diagram tabs: full-pane `MermaidBlock` (pan/zoom/fullscreen
  reused) with a Source/Diagram toggle in the tab toolbar.
- New `DiagramView.tsx` component wrapping MermaidBlock for full-pane use;
  tab `mode` extended with `'diagram'` alongside `'md' | 'code' | 'git'`.
- File association for `.mmd` added in electron-builder/Velopack config.
- Editing a `.mmd` in the built-in editor uses CodeMirror as today (plain
  text); Diagram view is the read mode.

**Real-time re-render on save:**

- New per-file watcher IPC: `fs:watchFile(path)` / `fs:unwatchFile(path)` /
  `fs:onFileChanged` in `electron/main.cjs` + `preload.cjs`, using
  `fs.watch` on the single file with a 150ms debounce (handles editor
  atomic-save rename/change event pairs; re-arm watcher if the inode is
  replaced).
- Renderer: every open tab registers a file watcher; on change, re-read
  only that file (`fs:read`), update ContentStore, re-render. Target
  save-to-render latency < 300ms.
- Folder-wide watcher (1200ms) is kept for tree refresh and non-open files;
  open-tab updates no longer wait for it.

**Reliability fixes (`MermaidBlock.tsx`):**

- Upgrade `mermaid` to latest 11.x.
- Render pipeline becomes a per-component async queue keyed by diagram id:
  a new render request supersedes an in-flight one; concurrent diagrams in
  one document cannot clobber each other's DOM/ids.
- On parse error:
  - Keep the last good SVG visible with a "stale — source has an error"
    badge.
  - Collapsible error panel below it showing mermaid's error message and
    the offending line (mermaid 11 error includes line info; fall back to
    raw message).
  - If no previous good render, show the error panel + raw source block
    instead of an empty box.
- Input normalization passes (applied only on retry after a failure, in
  order): existing HTML-entity rewrite → strip BOM, normalize CRLF→LF,
  replace smart quotes with ASCII quotes, trim trailing whitespace on the
  fence line.
- Error reporting to log file retained.

### Testing

- Unit (vitest): normalization passes (BOM/CRLF/smart quotes/entities),
  render-queue supersede behavior (mock mermaid), error-panel state
  transitions (good → bad keeps stale SVG; bad-first shows source).
- Integration: open `.mmd` fixture → diagram tab renders; modify fixture on
  disk → re-render fires (watch IPC mocked at electron-api boundary).

## Part 3 — UI/UX: Canva-style design system + 6 themes

### Current state

- 4 themes (`light`, `dark`, `cartoon-light`, `cartoon-dark`) in
  `src/lib/themes.ts` + CSS vars in `src/app/globals.css:14-75`, applied via
  `data-theme` on `<html>` + `.dark` class; persisted in localStorage; two
  favorite-theme quick-toggle slots.
- Bootstrap-free Tailwind UI; emoji/ad-hoc icons; mixed radii/shadows.

### Target state

**Design tokens (rebuilt `globals.css`):**

- Color: `--c-bg`, `--c-surface`, `--c-surface-2`, `--c-border`,
  `--c-text`, `--c-text-muted`, `--c-accent`, `--c-accent-2` (gradient
  endpoint), `--c-danger/-warn/-ok`.
- Shape: radius scale `--r-sm: 8px / --r-md: 12px / --r-lg: 16px`.
- Elevation: `--shadow-1/-2/-3` soft layered shadows (theme-tuned).
- Motion: `--ease` + 150/200/250ms duration tokens; respect
  `prefers-reduced-motion`.
- Existing UI-size (root rem scaling) and content-zoom mechanisms unchanged.

**Component restyle (no layout/structure change — sidebar, tabs, content,
bottom panel, settings modal all keep their behavior and props):**

- Pill buttons/inputs, rounded floating panels, gradient accent on active/
  primary states, refined tab bar (rounded tabs, animated active indicator),
  modern settings modal, custom slim scrollbars, hover-lift and fade
  micro-animations.
- Icons: replace emoji/ad-hoc icons with `lucide-react` throughout
  (sidebar, tabs, settings, toolbar, window controls keep themed style).
- Font: Inter (UI) + bundled Noto Sans Thai chain (kept); custom font
  feature unchanged.

**Themes (6):** defined in `src/lib/themes.ts` + `globals.css` blocks.

| id | mode | character |
|---|---|---|
| `light` (default light) | light | clean white, blue-violet accent |
| `dark` (default dark) | dark | deep slate, soft contrast |
| `aurora` | dark | purple-blue gradient accents, glass (blur) panels — Canva-like |
| `nord` | dark | arctic blue-grey (Nord palette) |
| `sepia` | light | warm cream paper, reading-optimized |
| `dracula` | dark | Dracula palette (purple/pink/cyan) |

- Cartoon themes removed; migration: stored `cartoon-light` → `light`,
  `cartoon-dark` → `dark` on load (one-line mapping in storage load).
- Per-theme mappings: mermaid theme + themeVariables, highlight.js style,
  titleBarOverlay colors (existing `window:setTitleBarColors` IPC), CodeMirror
  editor theme, xterm terminal palette — all driven from the theme definition
  so every surface matches.
- Theme switching, favorites (2 quick slots), persistence: mechanism
  unchanged.

### Testing

- Existing vitest suite must pass (components keep props/behavior).
- New unit tests: theme map completeness (every theme defines every token +
  mermaid/hljs/terminal mapping), cartoon→default migration mapping.
- Manual visual pass per theme across: sidebar, tabs, markdown view, code
  editor, diff view, terminal, search, settings, toasts, mermaid diagrams.

## Out of scope

- macOS/Linux packaging, code signing, layout/navigation changes (command
  palette, home screen), Vue/Bootstrap standards (this is a React project),
  mermaid render-as-you-type preview (may be a later phase).

## Risks

- Velopack migration bricking updates for current installs → mitigated by
  dual-artifact `v1.1.0` release + never uninstalling NSIS copy before the
  Velopack install verifies.
- mermaid 11.x upgrade behavior changes → covered by fixture-based unit
  tests before/after upgrade.
- Theme rewrite regressions in rarely-seen surfaces (toasts, context menus,
  drag overlays) → manual visual checklist per theme.
